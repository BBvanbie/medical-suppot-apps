# 医療情報システム安全管理ガイドライン ギャップ整理

更新日: 2026-04-19

## 目的

- `docs/Guideline/Overview.md`
- `docs/Guideline/Governance.md`
- `docs/Guideline/Management.md`
- `docs/Guideline/Control.md`
- `docs/Guideline/Q&A.md`

上記 5 文書と、現行の `docs/policies/`、`docs/operations/`、`docs/current-work.md`、既存 security / operations plan を照合し、現時点のプロジェクトで不足している準拠項目を今後の実装計画へ入れるための短い整理を残す。

## 結論

- 現行プロジェクトは認証、端末登録、監視、バックアップ、オフライン保護などの技術対策はかなり前進している。
- ただし、ガイドライン準拠という観点では、技術対策だけでは不十分で、`責任分界`、`リスク評価`、`規程 / 証跡`、`定期監査`、`非常時運用`、`委託 / クラウド管理`、`真正性 / 保存性` の整理がまだ弱い。
- 今後は `security / operations hardening` の後続ではなく、その上位にある `ガイドライン準拠ギャップ解消` として扱う。

## 現時点で足らない部分

- `責任分界の正式化`:
  医療機関側責任者、現場運用担当、開発保守担当、クラウド / 再委託先まで含めた責任分界表が repo にない。通常時、障害時、サイバー攻撃時での役割分担も未固定。
- `リスクアセスメント台帳`:
  患者情報、相談内容、通知、監査ログ、バックアップ、ローカル保存データの棚卸、重要度分類、保存場所、脅威、許容可否を残す台帳がない。
- `運用管理規程と証跡パッケージ`:
  認証手順、権限付与 / 削除、非常時手順、設定変更、教育、点検、監査、委託先確認の各記録をどこに残すかが未整理。
- `定期点検 / 監査運用`:
  監査ログと monitoring はあるが、誰が、どの頻度で、何をレビューし、逸脱時にどう是正するかの運用が不足している。
- `BCP / 非常時訓練`:
  incident / fail-safe / backup runbook はあるが、復旧優先順位、RTO/RPO、診療継続の代替手段、復元訓練記録、サイバー攻撃演習計画が弱い。
- `バックアップ世代 / 方式`:
  現状は回数と保持期間の整理が中心で、複数世代、複数方式、追記不能保管、ネットワーク分離保管、定期 restore drill の基準が不足している。
- `ネットワーク安全管理`:
  セキュアネットワーク、VPN / 外部接続、無線 LAN、ACL / FW、通信暗号化、監視点の構成図と運用責任が不足している。
- `端末 / 資産管理`:
  `EMS iPad`、`HOSPITAL PC`、`ADMIN` 端末の台帳、パッチ適用、マルウェア対策、廃棄 / 再配布、私物端末制御の標準化が不足している。
- `委託 / クラウド管理`:
  ベンダ選定基準、SLA、再委託管理、障害報告、ログ提供、監査協力、バックアップ責任を確認する checklist がない。
- `利用者 ID 棚卸と本人確認`:
  アカウント lifecycle はあるが、定期棚卸、不要 ID 無効化証跡、パスワード再発行時の本人確認台帳までは整っていない。
- `記録の真正性 / 見読性 / 保存性`:
  本システム内でどこまでを正式記録として扱うか、確定操作、更新履歴、代行入力、信頼できる時刻源、長期保存責務の整理が不足している。
- `外部保存 / 第三者提供整理`:
  クラウド backup、通知、将来の地域連携や外部 API 連携を含めた第三者提供 / 外部保存の整理が弱い。
- `教育 / 訓練`:
  管理者、現場利用者、保守担当ごとのセキュリティ教育、端末紛失対応訓練、インシデント初動訓練、受講 / 実施記録が不足している。
- `対外説明文書`:
  医療情報の扱い、外部保存、問い合わせ窓口、障害時の案内を外部説明できる文書が不足している。
- `製品機能としての未整理領域`:
  もし本システムを正式な診療録等の保存主体へ広げるなら、電子署名、確定操作、代行入力承認、改ざん防止、原状回復手順が追加で必要。

## 現時点の扱い

- 2026-04-14 時点では、本システムは `搬送調整支援システム` を主体として扱う。
- したがって、直近のガイドライン準拠対応は `責任分界 / リスク評価 / 規程 / 証跡 / 監査 / BCP / 委託管理 / 資産管理` を優先する。
- `真正性 / 見読性 / 保存性`、`電子署名`、`確定操作`、`代行入力承認` は、将来の `診療録等連携`、`電子カルテ連携` を扱う段階で再度計画化する。

## 搬送調整支援システムとして見た必要範囲

### P0: 直ちに必要

- `責任分界`
  - 医療機関側、運用担当、開発保守、クラウド事業者の責任を通常時 / 障害時で明確化する
- `リスク評価`
  - 患者情報、相談、通知、監査、オフライン保存、backup の資産棚卸と残リスク整理
- `認証 / 認可`
  - 利用者識別、権限最小化、不要 ID 棚卸、本人確認付き再発行、端末登録、MFA 方針
- `証跡 / 監査`
  - アクセスログ、操作ログ、security signal の記録と定期レビュー
- `BCP / backup / restore drill`
  - 診療継続を支える最低限の復旧計画、複数世代 backup、restore drill
- `委託 / クラウド管理`
  - 契約、SLA、再委託、ログ提供、障害報告、backup 責任

### P1: 早めに必要

- `端末 / 資産管理`
  - EMS iPad、HOSPITAL PC、ADMIN 端末の棚卸、再配布、紛失時対応
- `教育 / 訓練`
  - 紛失時、ログイン障害時、通知停止時、restore drill の教育記録
- `ネットワーク安全管理`
  - 外部接続、VPN、無線 LAN、通信暗号化、監視点、責任分界
- `対外説明`
  - 外部保存、問い合わせ、障害時の案内、患者情報の取扱い説明

### P2: 連携前に必要

- `外部保存 / 第三者提供整理`
  - 将来の地域連携、外部 API、電子カルテ連携に向けた責任分界と契約条件
- `連携時の認証 / 認可`
  - API 連携先、デバイス、ユーザー、通信経路の制限
- `監査 / 証跡の受け渡し`
  - どのログをどちらが持つか、障害時に誰が出すか

### P3: 今は原則不要

- `電子署名`
- `記録の確定操作`
- `入力者 / 確定者分離`
- `代行入力承認`
- `長期署名検証`
- `法定保存年限前提の真正性 / 見読性 / 保存性対策`

上記は本システム自体を `正式な診療録等の保存主体` にする場合のみ、本格対応が必要になる。

## やるべきことリスト

### 優先度 A

1. 責任分界、リスク評価、規程 / 証跡、委託 / クラウド確認の docs を導入組織向けに実名運用できる形へ仕上げる
2. ID 棚卸、監査 / 点検、BCP / restore drill、資産管理 / 教育の runbook を実運用記録付きで定着させる
3. backup の複数世代、複数方式、追記不能保管、分離保管の確認欄を実データで埋められる状態にする
4. 監査ログと security signal の定期レビューを月次運用へ乗せる

### 優先度 B

1. ネットワーク安全管理の責任分界と運用文書を追加する
2. 対外説明文書を追加し、外部保存と問い合わせ窓口を明文化する
3. 監視、backup、DB、通知連携の委託先台帳を作る

- `docs/operations/network-security-runbook.md` を更新し、`構成図 / 接続点 / FW・ACL / 無線 LAN / 外部接続 / 監視点 / 例外ルール` を運用文書へ反映した。
- `docs/policies/external-explanation-for-transport-coordination-system.md` を更新し、`外部保存 / 問い合わせ窓口 / 障害時案内 / 正式記録との違い` を説明文書へ反映した。
- `docs/operations/vendor-registry.md` を更新し、`契約 / SLA / 証跡 / 保存リージョン / 定期見直し / 未決事項管理` を台帳へ反映した。
- repo 側の文書雛形としては、優先度 B の `ネットワーク安全管理 / 対外説明 / 委託先台帳` まで初版を揃えた。

### 優先度 C

1. 診療録等連携の対象データ候補を整理する
2. 電子カルテ連携時の責任分界、正本、監査証跡の扱いを設計する
3. 正式記録化が必要になった場合の追加要件を別計画へ切り出す

## 計画への組み込み方

- Phase 1:
  `責任分界表`、`リスクアセスメント台帳`、`規程 / 証跡一覧`、`委託 / クラウド確認 checklist` を docs として先に固定する。
- Phase 2:
  `定期監査 / 点検`、`ID 棚卸`、`教育 / 訓練`、`BCP / restore drill`、`ネットワーク運用` の runbook を追加する。
- Phase 3:
  上記 docs で不足が確定した product / infra gap を、認証、監査、保存、代行入力、時刻管理、外部連携制御などの実装計画へ落とす。
  ただし `代行入力` や `正式記録の確定` は、診療録等連携フェーズまで保留する。

## 優先順位

- `優先度A`:
  責任分界、リスク評価、規程 / 証跡、認証 / 認可運用、定期監査、BCP / restore drill、委託 / クラウド管理
- `優先度B`:
  ネットワーク安全管理、端末 / 資産管理、教育 / 訓練、対外説明
- `優先度C`:
  診療録等連携、電子カルテ連携の前提整理
- `現時点では不要`:
  真正性 / 見読性 / 保存性、電子署名、確定操作、代行入力承認
