# 委託先台帳

更新日: 2026-04-19

## 目的

- 本システムで利用する外部事業者について、役割、責任分界、保持データ、障害時の連絡経路、再委託有無を整理する。
- 病院向け説明に出す項目と、内部管理にのみ使う項目を分けて管理する。
- 契約、SLA、証跡、再委託、保存場所、削除条件、定期見直しを 1 枚で追えるようにする。

## 対象カテゴリ

- アプリ実行基盤
- データベース基盤
- バックアップ保存先
- 監視 / 障害検知基盤
- 通知連携基盤

## 管理単位

- `1 事業者 1 行` を基本にする
- 同一事業者でも、役割や責任分界が異なる場合はカテゴリ別に行を分ける
- 病院向けに開示する内容と、内部管理だけで扱う内容を分けて記録する

## 内部で別途持つ情報

- 事業者名
- 契約番号
- 契約開始日 / 更新日
- 担当窓口
- 連絡先
- SLA
- 再委託先一覧
- 保存リージョン
- 削除 / 契約終了条件
- 初報時間
- 復旧目標
- 監査回答窓口
- 月次 / 四半期 / 年次の確認結果
- 未解決のリスク / 宿題

## データ区分の考え方

- `認証情報`
  ログイン、セッション、MFA、失敗記録に関する情報
- `端末登録情報`
  登録端末、端末識別子、登録状態に関する情報
- `患者基本情報`
  搬送調整に必要な患者の属性情報
- `搬送調整情報`
  事案、受入要請、搬送判断に関する情報
- `相談 / コメント履歴`
  EMS と HOSPITAL の調整会話や判断メモ
- `通知情報`
  通知生成、未読、配信結果に関する情報
- `監査ログ`
  操作履歴、権限逸脱、管理操作記録
- `監視イベント`
  API failure、security signal、notification failure などの検知情報
- `バックアップデータ`
  DB dump、restore 用データ、保持メタデータ

## 運用ルール

- 病院向け説明では、必要がある場合に `委託カテゴリ`、`役割`、`責任分界の概要` を開示する。
- 事業者名、契約番号、直接連絡先は内部管理情報として扱う。
- 再委託がある場合は、事前承認と責任分界確認を必須にする。
- 保存リージョン、削除条件、障害報告、証跡提供、再委託有無は契約または確認資料で裏取りする。
- `導入前 / 運用中 / 見直し中` の状態を持ち、未確認事項を残したまま本番運用へ持ち込まない。
- 病院向け開示可否が `条件付き可` のものは、説明文書に出す粒度を企画管理者が事前承認する。

## 定期見直し

| 頻度 | 見直し内容 | 主担当 |
|---|---|---|
| 月次 | 障害、backup failure、通知 failure、窓口変更 | システム運用担当 |
| 四半期 | SLA、再委託、保存リージョン、責任分界、契約差分 | 企画管理者 |
| 年次 | 契約更新、監査協力、削除条件、残リスク、継続利用可否 | 管理責任者 / 企画管理者 |

## 未決事項の残し方

- `要確認` の項目は、確認先、期限、担当者を別途チケットまたは導入台帳に残す。
- 本台帳には `未解決のリスク / 宿題` を 1 行で追記し、次回見直し時に閉じる。
- 契約終了や事業者変更時は、データ返却、削除証跡、アカウント停止、監査ログ保全の完了を確認する。
