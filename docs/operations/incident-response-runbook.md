# 障害時対応 Runbook

## 目的

- システム停止、通知停止、同期不整合などの障害時に、`ADMIN` と運用責任者がどの順で切り分けるかを固定する。
- 現状の監視画面、オフライン導線、通知運用を前提に、暫定運用へ切り替える判断を明文化する。

## このプロジェクトで想定する主要障害

- ログインできない
- `EMS` の病院検索や送信が失敗する
- `HOSPITAL` の受入要請一覧や相談が更新されない
- 通知ベルが増えない
- `Offline Queue` が詰まり続ける
- `ADMIN / 監視` で DB / API / backup 異常が出る

## 初動で必ず記録すること

- 発生時刻
- 発見者
- 影響ロール
- 影響画面
- 事案 ID / request ID / target ID があればその値
- 継続中か、一時的か
- 直前の操作

## 共通初動

1. `ADMIN / 監視` を開く
2. 影響範囲を `全体障害` と `部分障害` に切り分ける
3. 直近リリース、DB 障害、通知障害のどれに近いか判断する
4. 必要なら「手動更新での暫定運用」か「送信停止案内」へ切り替える
5. 連絡先に一次共有する

## 1. システム停止

### 典型症状

- `/login` が開かない
- 主要画面が 500 または空白
- 全ロールで一覧取得に失敗する

### 確認項目

- `DB 接続状態`
- `重要 API 失敗`
- 直近デプロイ有無
- アプリケーションプロセスの生存

### 一次対応

1. 書き込み系操作を一時停止するか判断する
2. 直近デプロイ起因なら [release-runbook.md](/C:/practice/medical-support-apps/docs/operations/release-runbook.md) に従ってロールバックを検討する
3. DB 起因なら復旧優先で進める
4. 復旧後に `EMS`、`HOSPITAL`、`ADMIN` の代表導線を確認する

## 2. 通知停止

### 典型症状

- 要請や相談は処理されているが通知ベルが増えない
- 相談コメント後に相手側で気づけない

### 確認項目

- `通知生成 / 配信失敗`
- 対象 API の失敗有無
- 対象ロールの偏り

### 一次対応

1. 影響ロールを特定する
2. 暫定運用として「一覧画面の手動更新」を案内する
3. 重要案件は電話併用を案内する
4. 復旧後に未達期間の案件を spot check する

## 3. 同期不整合 / オフライン問題

### 典型症状

- `EMS` で保存した内容がサーバーと合わない
- `Offline Queue` に conflict が残る
- 送信後の状態変化が画面へ反映されない

### 一次対応

1. 対象案件を特定する
2. `Offline Queue` の状態を確認する
3. 監査ログと最新 server 状態を確認する
4. `server を採用` か `local を採用して再保存` のどちらに寄せるか判断する

補足:

- 現在の初期方針は自動マージではなく手動解決
- request 状態や搬送判断系は server 正本を優先しやすい

## 4. ログイン障害

### 典型症状

- ロック中ユーザーが急増
- 特定ロールだけログインできない
- 一時パスワードや端末登録が進まない

### 一次対応

1. `ログイン失敗急増` を確認する
2. 共通の ID/PASS 配布ミスか、個別ロックかを分ける
3. 必要なら対象ユーザーに一時パスワード再発行を行う
4. 未登録端末問題なら端末登録ガイドを案内する

## 障害レベルの目安

- `重大`
  - 全ロール影響
  - ログイン不可
  - 要請送信不可
  - DB 接続異常
- `高`
  - EMS または HOSPITAL の主要導線停止
  - 通知全停止
- `中`
  - 特定ロール、特定画面のみ
  - TRAINING だけの不具合

## 復旧後確認

- ログイン可否
- `EMS` の病院検索と送信
- `HOSPITAL` の受入要請一覧と相談送受信
- `ADMIN / 監視` の異常収束
- 通知件数の増加
- 主要 API エラーの解消

## 恒久対応へ残すこと

- 原因
- 再発防止策
- 暫定運用の有無
- 利用者告知が必要か
- 関連 runbook 更新の要否
