# 救急搬送支援システム 正式記録 scope decision

作成日: 2026-04-14

## 論点

本システムを、`搬送調整支援システム` として扱うのか、または `診療録等の正式保存主体を含むシステム` として扱うのかを決める必要がある。

## 現状確認

- 現行仕様は `事案作成`、`病院検索`、`受入要請`、`相談`、`搬送判断`、`通知`、`監査` を中心に構成されている。
- 既存 docs とコードには、電子署名、確定操作、代行入力承認、長期署名検証、法定保存年限を前提にした構造はない。
- 一方で患者情報、相談履歴、搬送判断を扱うため、医療情報安全管理ガイドラインの一般的な安全管理は当然必要である。

## 選択肢

### 案 A: 搬送調整支援システムとして扱う

- 位置付け:
  正式な診療録等の保存主体ではない
- 必要なもの:
  認証、認可、監査、BCP、委託管理、バックアップ、運用証跡
- 不要または後回し:
  電子署名、記録の確定、代行入力承認、法定保存年限前提の長期署名検証
- 利点:
  現行プロダクトと整合しやすい
- 欠点:
  正式記録や法定文書を将来扱う場合は再設計が必要

### 案 B: 正式な診療録等の保存主体も含むシステムとして扱う

- 位置付け:
  一部または全部を法定保存対象の正式記録として扱う
- 追加で必要なもの:
  記録の確定操作、入力者 / 確定者分離、代行入力管理、電子署名、タイムスタンプ、保存年限、長期検証、見読性要件、原状回復手順
- 利点:
  ガイドラインの真正性 / 見読性 / 保存性に正面から対応できる
- 欠点:
  現行アーキテクチャ、UI、運用の大幅見直しが必要

## 推奨

- 現段階では `案 A` を推奨する。
- 理由:
  現行システムは搬送調整、相談、受入運用の支援を主目的としており、正式な診療録システムとしての要件まではまだ入っていないため。

## 2026-04-14 決定

- 当面は `搬送調整支援システム` を主体として扱う。
- 今後の流れは以下とする。

1. まず搬送調整システムとして安全管理、運用証跡、委託管理、BCP を固める
2. 次段で診療録等への連携要件を整理する
3. その後、電子カルテ連携要件を別計画で扱う

- したがって、現時点では `電子署名`、`確定操作`、`代行入力承認`、`長期署名検証` は直近実装テーマに入れない。
- ただし、将来連携に備えて以下は後続テーマとして維持する。
  - 連携対象データの定義
  - 連携時の責任分界
  - 監査 / 証跡の受け渡し
  - 正式記録化が必要になった場合の拡張余地

## この判断で直近進められること

- ガイドライン一般部分:
  責任分界、リスク評価、証跡、監査、BCP、委託管理
- 運用強化:
  ID 棚卸、資産管理、教育、restore drill
- infra / security 強化:
  backup 複線化、ログ保管、通知経路、ネットワーク責任分界
- product の次段テーマ:
  診療録等連携、電子カルテ連携の前提整理

## 案 B を選ぶ場合に直ちに必要な追加テーマ

1. 正式記録の対象データ定義
2. 確定操作 UI / API
3. 代行入力と承認フロー
4. 電子署名 / タイムスタンプ方式
5. 長期保存と検証方式
