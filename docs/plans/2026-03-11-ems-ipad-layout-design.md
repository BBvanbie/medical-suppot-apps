# EMS iPad Layout Design

**Date:** 2026-03-11

## Goal

A側画面を `1180x820` を基準とした iPad Air 4 想定レイアウトへ最適化し、事案作成画面の覚知情報行の崩れを解消する。あわせて、iPad Pro 以下での文字過大表示を抑え、A側のみで表示スケールを制御できるようにする。

## Scope

- A側のみ対応
- HP側のレイアウトや文字サイズは変更しない
- 完全レスポンシブ化は行わない
- `1180x820` を基準にしつつ、やや狭い幅を吸収する最小限の縮小対応を入れる

## Problems

### 1. 覚知情報行の崩れ

事案作成画面の `覚知日付 / 覚知時間 / 指令先住所` の入力行が iPad 幅で干渉し、入力欄が被る。`date / time / text` 入力で高さも揃っていないため、視認性が落ちている。

### 2. 文字サイズが大きすぎる

iPad 表示で、A側の見出し、ラベル、一覧セル、ボタン文言が大きく、2行化や3行化が増えている。固定端末運用なのに PC 寄りの文字スケールになっている。

## Alternatives

### A. A側だけ固定端末最適化

- `1180x820` を前提に A側画面を整える
- iPad Pro 以下だけ既定で 1 段階縮小
- 表示設定で上書き可能にする

Pros:

- 現場端末前提に最適化しやすい
- 改修範囲を A側だけに限定できる
- 今の課題に対して最短で効く

Cons:

- 将来 A側の利用端末が増えると追加調整が必要

### B. 完全レスポンシブ対応

- 画面幅ごとに全体レイアウトを組み直す

Pros:

- 端末汎用性は高い

Cons:

- 現要件に対して過剰
- A側フォーム、一覧、モーダルまで広く影響する
- HP側を触らない方針と相性が悪い

### C. 設定変更のみ

- 既定は変えず、設定で文字サイズ変更だけ可能にする

Pros:

- 実装は軽い

Cons:

- 初期表示の崩れが直らない
- 現場で設定前提になる

## Recommendation

案 A を採用する。A側は固定端末運用のため、`1180x820` 基準の最適化がもっとも合理的。完全レスポンシブ化はコスト過多で、設定のみでは初期状態の問題を解消できない。

## Design

### 1. A側共通の表示スケールを導入

A側だけに効く表示スケールを用意する。概念上は `compact / default / large` の 3 段階とし、既定値は端末幅で決める。

- `iPad Pro 以下` は `compact`
- それより広い A側表示は `default`
- 表示設定が保存されていれば、その設定値を優先

このスケールは CSS 変数または `data-*` 属性で A側ルートへ渡し、個別画面はその値を参照して文字サイズと入力高さを調整する。

### 2. 事案作成画面の覚知情報行を固定端末向けに組み直す

現在の `12 カラム` ベースのままでは iPad 幅で競合しやすいため、覚知情報だけは独立したレイアウトルールにする。

想定:

- `覚知日付`: 固定幅寄り
- `覚知時間`: 固定幅寄り
- `指令先住所`: 残り幅を広く使う

入力欄の高さは `date / time / text` をすべて同じ高さに揃える。ラベルの行高も統一し、iPad 幅で 1 行のまま収まるようにする。

### 3. A側の文字サイズを縮小

対象:

- 一覧見出し
- テーブルセル
- フォームラベル
- ボタン
- 補助説明文

ただし次は縮小しすぎない。

- 警告文
- ステータスバッジ
- 主要アクションボタン

`compact` では `text-sm -> text-xs`、`text-xs -> さらに小さい変数値` のように段階的に落とし、改行を減らす。

### 4. 設定画面との接続

既存の `救急隊 > 表示設定` の文字サイズ設定を A側画面に反映させる。これにより、既定は端末幅で最適化しつつ、利用者ごとの見やすさ調整も可能にする。

## Affected Areas

- `components/cases/CaseFormPage.tsx`
- `components/cases/CaseSearchTable.tsx`
- A側シェルまたは A側共通レイアウト
- `components/settings/EmsDisplaySettingsForm.tsx`
- A側表示設定読込ロジック

## Error Handling

- 表示設定の取得に失敗した場合は既定値を使う
- 既定値は `iPad Pro 以下 => compact`、それ以外 => `default`
- レイアウト崩れを防ぐため、覚知情報行は設定有無に関係なく高さを固定する

## Validation

- `1180x820` 相当で事案作成画面の覚知情報行が重ならない
- 日付、時間、住所の入力高さが一致する
- A側一覧・フォームで 2 行化が減る
- 表示設定の変更で A側の文字サイズが切り替わる
- HP側の見た目は変わらない

## Risks

- 既存の `text-xs / text-sm` 直書きが多い場合、A側全体で完全統一するには段階的な整理が必要
- 文字縮小の影響でタップ領域まで小さくしないよう、高さと余白は別管理にする
