# 外部説明用 HTML スライド Design

作成日: 2026-04-12

## 1. 目的

`救急搬送支援システム` を外部向けに説明する単一 HTML スライドを作る。

狙い:

- PoC 実施の承認につなげる
- 導入検討の次回打ち合わせにつなげる
- 現在のプロトタイプ完成度を伝える
- 予算、協力体制、実証フィールドの確保につなげる

対象:

- 病院、消防、自治体などの現場 / 導入判断者
- 経営層、行政担当、非エンジニアの意思決定者

## 2. 成果物

- 単一 HTML スライド
- ブラウザで直接開ける
- ネットワーク接続なしでも本文とレイアウトが成立する
- キーボード操作で前後移動できる
- 発表者メモは入れない
- 実アプリ画面スクリーンショットを使う
- 導入先名、地域名、病院名は仮名で扱う

想定配置:

- HTML: `docs/presentations/emergency-transport-support-system.html`
- assets: `docs/presentations/assets/`

## 3. 表現方針

採用方針:

- ハイブリッド型
  - 前半: 課題と導入価値を図解
  - 中盤: 実画面スクリーンショットで完成度を訴求
  - 後半: PoC 計画、期待効果、次段階を整理

トーン:

- 公的機関にも出せる堅実さ
- プロダクトとしての完成度が伝わる見栄え
- 医療 / 行政向けの信頼感
- 過度なスタートアップ風や派手な演出は避ける

視覚方針:

- 白基調
- deep blue + teal + emerald + amber を意味別に使う
- 画面スクリーンショットは 1 スライド 1-2 枚まで
- 図解は大きく、本文は短く
- タイトルは日本語中心
- 英語の装飾見出しは補助に限定する

## 4. 使用する skill / 参照

主参照:

- repo-local `docs-writer`
- repo-local `frontend-ui`
- global `visual-explainer`

`visual-explainer` 採用理由:

- HTML slide deck / visual explanation に向く
- 2026-04-12 に `nicobailon/visual-explainer@visual-explainer` をインストール済み
- GitHub stars が多い公開 repo として確認し、候補の中で信頼性が高い
- 使用前に `SKILL.md` を確認済み

注意:

- skill の共有 / deploy 機能は使わない
- 外部 CDN は必要最小限。オフライン閲覧性を優先する
- AI っぽい紫 / ネオン / gradient text / emoji header は避ける

## 5. スライド構成

20 枚、20-30 分想定。

### 1. 表紙

見出し:

- 救急搬送支援システム

本文:

- EMS・病院・管理者をつなぎ、搬送先選定と受入判断を同じ事案情報で進めるための業務支援システム。

ビジュアル:

- EMS、病院、管理者、指令を線でつなぐ全体図

### 2. なぜ今必要か

見出し:

- 搬送先選定は、現場判断と情報共有の負荷が集中する

本文:

- 電話連絡と手元記録に依存すると、状況変化や返信履歴を追いにくい
- EMS と病院で見ている情報が分断されやすい
- 管理者が滞留、重複、遅延を把握しにくい
- 訓練と本番データを分けた運用が必要

### 3. 現場で起きている課題

見出し:

- それぞれの立場で、判断材料が分散している

本文:

- EMS: 患者情報、候補病院、返信状況、相談内容を短時間で比較する必要がある
- HOSPITAL: 自院宛の依頼を見落とさず、患者概要と次アクションをすぐ確認したい
- ADMIN: 個別介入ではなく、滞留・認証・端末・監査を横断して見たい

### 4. 解決の方向性

見出し:

- 事案を中心に、関係者が同じ状態を追えるようにする

本文:

- 患者サマリー、病院検索、受入要請、相談、搬送判断を 1 つの事案に集約
- 受入状態をステータスとして共有
- 通知と履歴で「いつ、誰が、何をしたか」を追跡
- 管理者は全体監視と運用統制に集中

### 5. システム全体像

見出し:

- 4 つのロールで搬送調整を支える

本文:

- EMS: 入力、検索、送信、相談、搬送判断
- HOSPITAL: 自院宛依頼、患者確認、要相談、受入可否
- ADMIN: 監視、管理、監査、訓練運用
- DISPATCH: 指令起票、指令由来事案の確認

### 6. 導入後の業務フロー

見出し:

- 事案作成から搬送判断までを一連の流れで扱う

本文:

1. DISPATCH / EMS が事案を作成
2. EMS が患者情報とバイタルを入力
3. EMS が候補病院を検索
4. HOSPITAL へ受入要請を送信
5. HOSPITAL が受入可否または要相談を返す
6. EMS が搬送決定 / 辞退を行う

### 7. EMS 画面: 現場入力

見出し:

- EMS は現場情報を短時間で整理できる

本文:

- 患者基本情報、現場情報、主訴、バイタル、所見を入力
- 入力内容は患者サマリーとして整理
- iPad 横画面を前提に、比較と操作を近接配置

スクリーンショット候補:

- 事案作成画面
- 患者サマリー画面

### 8. EMS 画面: 病院検索と送信

見出し:

- 候補病院を比較し、複数施設へ受入要請を送信

本文:

- 病院検索結果を一覧で比較
- score と理由、診療科、距離、状態を確認
- 複数病院へ送信し、履歴として残す

スクリーンショット候補:

- 病院検索結果
- 送信履歴

### 9. HOSPITAL 画面: 自院宛依頼の確認

見出し:

- 病院は自院宛の依頼だけを一覧で確認

本文:

- 未読、既読、要相談、受入可能、受入不可を状態で把握
- 患者サマリーと選定履歴を同じ導線で確認
- 自院として次に押せる操作を明確化

スクリーンショット候補:

- HOSPITAL 受入要請一覧
- detail modal / detail panel

### 10. HOSPITAL 画面: 要相談と受入可否

見出し:

- 相談内容と回答履歴を同じ事案上に残す

本文:

- EMS と HOSPITAL のコメントを時系列で管理
- 要相談から受入可能 / 受入不可へ遷移
- 受入不可時は理由入力と電話連絡導線を用意

スクリーンショット候補:

- 相談チャット
- 受入不可理由 dialog

### 11. ADMIN 画面: 監視と管理

見出し:

- 管理者は現場介入ではなく、監視と統制に集中

本文:

- 滞留案件、API 失敗、通知失敗、不正操作兆候を確認
- ユーザー、端末、組織、病院、救急隊を管理
- 監査ログで操作履歴を追跡

スクリーンショット候補:

- ADMIN ホーム
- ADMIN 監視
- 端末管理

### 12. 状態管理と通知

見出し:

- 「今どうなっているか」をステータスと通知で共有

本文:

- 未読、既読、要相談、受入可能、受入不可、搬送決定を明確化
- 状態変化、相談コメント、搬送判断を通知
- dedupe と ack で通知の重複や確認漏れを抑制

### 13. 安全性: 認可と状態遷移の保護

見出し:

- 誤操作や越境アクセスを防ぐための制御を実装

本文:

- EMS は自隊事案のみ確認・操作
- HOSPITAL は自院宛 target のみ操作
- 他隊 / 他院アクセス拒否
- 二重搬送決定防止
- 終端状態からの再遷移拒否

### 14. 認証・端末運用

見出し:

- 本番運用を見据えた認証と端末管理

本文:

- ID / password
- EMS / HOSPITAL の WebAuthn MFA
- 登録コードによる端末登録
- 一時パスワード発行と強制変更
- ログイン失敗ロック

スクリーンショット候補:

- login
- device registration
- admin devices
- change password

### 15. オフライン・訓練・監視

見出し:

- 現場停止を避け、訓練と本番を分けて運用できる

本文:

- EMS のオフラインキュー、再送、競合確認
- LIVE / TRAINING mode の分離
- 本番 analytics へ training data を混入させない
- health check、監視画面、backup run report を用意

### 16. PoC で確認できること

見出し:

- PoC では、現場フローと運用保全を一連で確認できる

本文:

- EMS の事案入力から病院検索、受入要請送信まで
- HOSPITAL の依頼確認、要相談、受入可否
- EMS の搬送決定 / 辞退
- ADMIN の監視、端末、ユーザー、監査ログ
- TRAINING mode で本番データと分離した検証

### 17. PoC の進め方

見出し:

- 小さく始め、運用課題を確認しながら段階的に広げる

本文:

- 対象: A消防本部、B総合病院、C市医療圏などの仮想または限定参加
- 期間: 2-4週間を想定
- 方法: TRAINING mode で実運用と分離
- 確認: 入力負荷、病院応答、通知、履歴追跡、管理者確認
- 成果物: 改善点、運用ルール、次段階スコープ

### 18. 期待される効果

見出し:

- 搬送調整の「見えない負荷」を記録と状態で軽くする

本文:

- EMS: 候補病院、返信状況、相談履歴を比較しやすくなる
- HOSPITAL: 自院宛依頼と次アクションが明確になる
- ADMIN: 滞留、失敗、不正兆候、端末運用を把握しやすくなる
- 組織全体: 記録追跡、訓練、本番分離、改善サイクルを作れる

### 19. 次段階で強化すること

見出し:

- PoC 後に、本番運用へ向けた強化を進める

本文:

- 端末 fingerprint / 登録情報の強化
- backup job 自動連携
- 外部監視 / 通知経路の接続
- `hospitalCache` 暗号化判断
- 10000件以上の性能確認
- 導入先運用に合わせた帳票・権限・マスタ調整

### 20. クロージング

見出し:

- 実証に向けて確認したいこと

本文:

- 参加ロール: EMS、HOSPITAL、ADMIN、DISPATCH のどこまで含めるか
- 対象業務: 事案入力、病院検索、受入要請、相談、搬送判断
- 検証環境: TRAINING mode、仮名データ、限定参加
- 評価観点: 操作時間、見落とし、履歴追跡、管理負荷
- 次回: デモ環境確認と PoC 範囲の確定

## 6. スクリーンショット方針

実スクリーンショットを使う。

必須候補:

- EMS 事案作成
- EMS 病院検索
- EMS 送信履歴 / 搬送判断
- HOSPITAL 受入要請一覧
- HOSPITAL 相談 / 受入可否
- ADMIN ホーム
- ADMIN 監視
- ADMIN 端末管理
- Offline Queue
- Training banner

取得条件:

- ローカル app が起動済みであること
- TRAINING / demo data を使うこと
- 個人名、電話番号、住所、実病院名が入る場合はマスクすること

初期 HTML では screenshot slot を用意し、画像が未取得の場合も資料として成立するようにする。

## 7. UI / 実装要件

- 1 slide = 100dvh
- 縦スクロールではなく slide navigation
- 左右キー、Space、PageUp / PageDown 対応
- 画面下に progress indicator
- slide number 表示
- 印刷時は slide ごとに page break
- 16:9 presentation を主対象にする
- 1280px 以上で最適化
- 小さい画面では縮小して読めるようにする
- 発表者メモなし

## 8. 完了条件

- HTML がブラウザで開ける
- 20 枚の slide が揃っている
- 実画面 screenshot slot がある
- 画像未取得でも broken image にならない
- 日本語が文字化けしない
- `npm run check` が通る
- 可能ならブラウザで表示確認し、主要 slide が崩れていないことを確認する
