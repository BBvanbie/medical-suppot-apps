# docs-writer

## purpose

- 実装と運用の前提を、次の作業者が迷わない形で残す。

## use this skill when

- `README.md`、`docs/`、`MIGRATION_NOTES.md`、plan 文書を更新するとき
- 新しい作業手順、設計判断、移植方針を記録するとき
- 既存ドキュメントの齟齬を減らしたいとき

## do not use this skill when

- docs 更新が不要な局所コード修正だけを行うとき
- 技術判断より実装そのものが主作業のとき

## workflow

1. 既存 docs と現在の実装を照合する。
2. 読者をこのプロジェクトの次の実装者に固定する。
3. 目的、運用、制約、確認手順を優先して書く。
4. 古い情報を増やさないよう、重複より参照を優先する。

## output format

- 目的
- 変更内容
- 運用方法
- 注意点

## quality bar

- 実装と矛盾しないこと
- 抽象的でなく、実際のコマンドやパスを含むこと
- 冗長な説明より運用可能性を優先すること

## project-specific notes

- 新しい設計/実装作業は `docs/plans/` の命名規則に合わせる。
- UI や運用ドキュメントは日本語中心で維持する。
