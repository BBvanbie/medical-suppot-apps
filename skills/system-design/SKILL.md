# system-design

## purpose

- 要件を曖昧なまま実装に流さず、設計判断を小さく明確に固める。

## use this skill when

- 新機能、移植、構成変更、複数ファイルにまたがる変更を始めるとき
- 既存 docs や実装を読んで方式比較をしたいとき
- 仕様差分、責務分離、段階導入方針を整理したいとき

## do not use this skill when

- 単純な文言修正や既存仕様どおりの局所修正だけを行うとき
- 既に承認済みの設計に沿って実装だけ進めるとき

## workflow

1. 関連コード、`README.md`、`docs/`、最近の plan を確認する。
2. 制約、影響範囲、成功条件を整理する。
3. 2-3案あれば比較し、推奨案を明示する。
4. 主要な設計判断、非目標、検証方針を残す。

## output format

- 結論
- 前提と制約
- 方式比較
- 推奨設計
- 影響ファイル/検証方針

## quality bar

- 既存構成との整合があること
- 一般論ではなくこのリポジトリの構成に触れていること
- 実装時に迷わない粒度になっていること

## project-specific notes

- 役割別 UI と API の分離があるため、`EMS`、`HOSPITAL`、`ADMIN` の影響範囲を必ず分けて考える。
- 設計変更時は `docs/plans/` へ記録する。
